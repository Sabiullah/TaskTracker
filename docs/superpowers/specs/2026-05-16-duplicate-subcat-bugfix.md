# Duplicate Sub-Cat Plan Bugfix — May 16 2026 (round 2)

## Symptom

A goal materialised TWO sets of children for the same-looking subcategory
("Data Collection - Stock Report"): one set on Mondays (the correct
cadence), another on Fridays (a stale Onetime+5 cadence). Editing a
wrong-cadence row's date to a Monday also created a duplicate: the
edited date appeared as a new row while the original date came back on
the next reload.

## Root cause

Two distinct subcategory masters were named "Stock Report" — one under
the user's intended main category (`recurrence=Weekly, target_day=1`)
and another under a same-named-but-different main category (or under a
different parent visible to the user via org-sharing) with
`recurrence=Onetime, target_day=5` or `recurrence=Weekly,
target_day=5`. The frontend's save paths resolved sub-cat names against
the global `catMasters` list without filtering by the goal's chosen
main category, so:

- `buildPlansPayload` could pick the wrong master and POST its UID.
- `handleAddPlan` (in `+ Add subtask`) could pick the wrong master.
- The goal ended up with plans for **both** Stock subcategory UIDs.
  Because `TaskSubcategoryPlan.unique_together = (main_task,
  subcategory)` is by UID, the two plans coexist — no constraint to
  collapse them.

The materialiser then emitted children for **each** plan, and when the
user edited one plan's child's date, the materialiser on next view
re-created the original date for the other plan because the dedupe
`(category_id, target_date)` set didn't see it.

## Fix

`subTemplatesByMain` is now keyed by **parent UID** (`subTemplatesByMainUid`)
rather than parent name. The form derives `selectedMainUid` from
`form.category` (name → first-match UID, same deterministic rule as
`categoryUidByName` so the save path agrees with the preview).

Every sub-cat lookup now flows through UIDs:

- `SubtaskItem` gains an optional `subcategoryUid`. Rows generated by
  `buildSubsFromTemplate` and `dtoToTaskAsSub` pin it to the exact
  master that produced them.
- `buildPlansPayload` prefers `row.subcategoryUid` over any name lookup.
  Fallback name lookups are filtered by the goal's main UID.
- `handleAddPlan` filters its name → UID resolution by the selected
  main UID.
- `handleRecurrenceChange` resolves `target_day` from the row's pinned
  sub-cat UID first.

Result: a newly-created goal can only ever attach to the sub-cat
masters under the chosen main, regardless of how many same-named
sub-cats exist elsewhere.

## Repairing the existing goal #5222

The duplicate plan is already in the DB. To clean it up:

1. Open Edit Goal #5222.
2. The Stock Report rows include both a Monday set (correct) and a
   Friday set (stale). Click the **X** on one of the Friday rows
   (e.g. 01/05 or 08/05).
3. Confirm the prompt. Because the view month is May 2026 — the same
   month as the plan's `active_from_month` — `cap_plan` hard-deletes
   the entire stale plan plus all its uncompleted children (every
   Friday across all months, not just the visible ones).
4. The remaining Stock rows are the correct Monday cadence.

Future goal creations under "Data Collection - Stock Report" (or any
same-named subcategory) will pick the correct master based on the
chosen main category — no more duplicate plans.
